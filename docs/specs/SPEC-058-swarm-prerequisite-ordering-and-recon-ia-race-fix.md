## 🎭 驗收場景（Acceptance Scenarios）

> 使用 Gherkin 語法描述使用者情境。每個場景對應測試矩陣中的一行或多行。
> **非 trivial 任務必填**。AI 從場景自動產生測試骨架。

```gherkin
Feature: {SPEC 標題}
  作為 {角色}
  我想要 {行為}
  以便 {價值}

  Background:
    Given {共通前置條件}

  # --- 正向場景 ---

  Scenario: S1 - {正向場景描述}
    Given {前置條件}
    When {使用者操作}
    Then {預期結果}
    And {額外驗證}

  # --- 負向場景 ---

  Scenario: S2 - {負向場景描述}
    Given {前置條件}
    When {觸發錯誤的操作}
    Then {預期錯誤處理}

  # --- 邊界場景 ---

  Scenario Outline: S3 - {邊界場景描述}
    When {帶參數的操作 "<param>"}
    Then 回應狀態碼為 <status>

    Examples:
      | param    | status |
      | {邊界值1} | {結果1} |
      | {邊界值2} | {結果2} |
```

> **場景撰寫規則**：
> - 每個場景必須有至少 1 個 `Then` 斷言（品質門 G2 會檢查）
> - 每個場景必須有 `Given` 或 `Background` 前置條件
> - 場景名稱必須具體（禁止 "it works"、"正常" 等模糊名稱）
> - Bug 修復的場景必須包含一個「重現原 bug」的負向場景

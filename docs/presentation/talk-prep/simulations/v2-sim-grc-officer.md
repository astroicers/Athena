## 詳細分析

### 我會帶回去做的三件事

**第一件 — 寫進年度資安威脅趨勢報告。** 我的「AI 在資安的應用」章節原本只有兩頁泛論，現在我會多一頁實證：引用「20 分鐘 / 全自動 / 全滅 / 100% 信心可解釋」這組數字（slide 54），標註資料來源是 CYBERSEC 2026 在地紅隊團隊現場示範，不是國外二手翻譯。然後在威脅 landscape 章節加一張表複製 slide 60 的三個 APT 案例，特別標註 Volt Typhoon 與台海評估。處長下週會問「AI 攻擊我們有沒有準備」— 我的答案就會圍繞這兩張投影片。

**第二件 — 下個月 12 份下屬機關資安計畫書的稽核項目。** 我會新增三個檢核點，直接借用 slide 61 那三個問題：(1) 機關是否能同時掌握雲端與地端攻擊面？(2) 事件應變程序是否涵蓋 AD、Entra ID、M365、雲端密鑰的憑證與 token 路徑？(3) 事件應變時間目標（MTTR）是否與當前自動化攻擊速度相符？第三點特別重要 — 我們現在很多機關的應變 SLA 還停在「24 小時內回報」，而講者展示的攻擊在 20 分鐘內就結束了。

**第三件 — 簡報處長 / 上級的 visual。** Slide 54 的 Mission Complete 那四格、Slide 5（Alex 段）的 blast radius 兩排（從初始入侵 → AD 立足點 → DA → 混合身分 → Azure 租戶 → M365 → Key Vault → 跨雲），就是處長級主管看一眼就懂的圖。我會把這兩張放進部會的內部簡報，搭配三個 APT 案例當情境說明。

### 為什麼 A / B / C / D 是這個分數

**A = 3.5。** 整場 30 分鐘對我這種非技術背景的稽核員來說，技術段落比例太高。Slide 13–20 的引擎內部、Slide 41–48 的 AD 攻擊細節，加起來大概 8 到 10 分鐘對我幾乎是無效時間。如果開場、Mission Complete tile、第二位講者的 APT 案例與三個問題能再展開一點，並且加上「對應 NIST CSF 哪幾個 function」「資安法第幾條」「關基保護指引哪個面向」這類治理錨點，我會給 4.5。

**B = 4。** 我學到三件以前不知道的事：(1) AI 紅隊工具已經能做到全自動執行（不只是輔助 prompt），這對我威脅評估是新基準；(2) 混合身分認證的攻擊面（slide 56–57）— 我以前審計畫書只會問「有沒有 MFA」「有沒有 Entra ID 條件存取」，從來沒想過 Entra Connect 本身是控制面；(3) Storm-0558 跟 Midnight Blizzard 的細節 — 以前只知道有事件，不知道 Storm-0558 是 MSA 簽章金鑰外洩、Midnight Blizzard 是舊測試租戶被打進去。

**C = 3。** 攻擊技術細節（AS-REP、ADCS ESC1、secretsdump）對我的日常工作距離有點遠 — 我不會去看 KDC 設定、不會去讀 certipy 輸出。但治理層的訊號（『AD 設定錯誤』、『修了沒落地』、『治理放大器』、混合身分風險）跟我每天在做的事完全對得上。可惜這部分講者沒有用稽核員聽得懂的語言展開，所以分數沒法給更高。

**D = 4。** 我走出去手上有具體可以做的東西：年度報告兩段、稽核項目三條、處長簡報兩張圖。這比我之前去聽的好幾場 AI 主題場次都實在 — 多數場次離開時我只記得「AI 很強很危險」，但這次我有可以放進公文的素材。沒有給 5 是因為這些都還需要我自己回去翻成稽核語言、找到法規對應條文，講者沒有幫我做這個 last mile。

### Harry 段我的感受

開場的 Mission Briefing（slide 5）我看到「OPERATION / 24 張 / WEB01 → DC-01 → ACCT-DB / ~15 min」就有點警惕 — 24 張投影片講一條攻擊鏈，我大概率會在中間放空。事實上後來確實是這樣。

C5ISR 那段（slide 8, 10, 11）我意外地能跟上，因為他用了「軍事八十年的學費」這個 framing — 我們公部門的人對「指揮、控制、情報、通訊」這套語彙是熟悉的，跟我們行政體系的「指揮中心」、「應變小組」、「通報程序」是同一套思維。Slide 11 的「C5ISR → Athena 對應表」我有抄下來，雖然我不知道 BloodHound 跟 nmap 是什麼，但「情報分析 → Facts DB」、「執行控制 → 信心值 × 風險門檻」這個對應我看得懂 — 這就是我們稽核時會問的「決策依據」、「風險評估」。

中間 slide 13–20 的引擎內部我幾乎全程放空 — JSON、composite confidence、Schema、prompt injection、dynamic routing。我只在最後注意到 slide 20 提到「prompt injection via MCP description」，因為「allowlist」這個詞我懂 — 那是稽核項目的語言。

實戰段（slide 35–54）我承認大概只懂 40%。我看得懂時間戳記在跑、看得懂三台機器一台一台變紅、看得懂最後 Mission Complete 那四個數字。但 AS-REP Roasting 的 padata 解釋、ADCS ESC1 的三個條件、hashcat 線上對離線的差別 — 這些對我就是「攻擊技術之一」。我能帶走的只有「強密碼擋不住、AD 設定錯誤是主要破口」這個治理結論（slide 33 「五條，沒有一條跟密碼有關」、slide 34 「全部強密碼，密碼噴灑全敗」），這正是我下次審計畫書時可以拿來反問機關「你們的 AD 設定錯誤掃描多久一次」的稽核切入點。

### Alex 段我的感受

第二位講者只有 10 分鐘，但對我這種非技術背景的人來說，CP 值高過第一位的 20 分鐘。

開場的「Domain Admin → ?」（slide 55）我立刻聽進去 — 這就是我每天在公文裡看到的盲點：很多單位以為 AD 鞏固好了就安全，但他們的 M365、Azure 跑的是混合身分認證，AD 一倒整個雲端跟著倒。slide 56 的兩欄對比（地端 AD vs Entra ID/Azure/M365）就是我下個月稽核時可以直接反問機關的問題：「你們的真正資產住在哪？AD 還是雲？兩邊有沒有用同一條身分驗證鏈？」

Slide 57 的 C5ISR 延伸表（地端 vs 雲端）我有抄。「Comms：SSH/SMB/Kerberos/LDAP vs Access token/Refresh token/PRT/SAS」這行我特別有感 — 我們的稽核問卷現在還在問 SMB 簽章、LDAP 加密，但完全沒有 token 路徑的稽核項目。這是我接下來幾個月的功課。

slide 58 那段 flAWS.cloud 真實 log — 我知道對技術聽眾這是亮點（rec_id、ooda_iteration_id、實證），但對我來說那段 JSON 又是天書，我跳過了。我反而比較記得他講「不是更快，是更聰明」這句結論。

slide 60 三個 APT 案例是我這場最大的 takeaway。Storm-0558（MSA 簽章金鑰）、Midnight Blizzard（舊測試租戶）、Volt Typhoon（LOTL，台海評估）— 這三個案例的條件式描述格式（事件 → 攻擊路徑 → 後果），就是我年度報告威脅 landscape 章節要的格式。Volt Typhoon 跟台海連結這句話對我這個位置的人來說特別有重量 — 這不只是技術威脅，是國安議題。

slide 61 三個防守者問題我用紅筆抄了。這三個問題用的語言是我聽得懂的（紅隊、SOC、事件應變），不是工程師術語。我下個月稽核就用這三條當開場提問。

### 我希望這場 talk 多 / 少做的

**多做：** 我希望最後 5 分鐘能有「對應到治理框架」的一張投影片。具體來說：(1) 把這場展示的攻擊鏈對應到 NIST CSF 的 Identify / Protect / Detect / Respond / Recover 五大 function，告訴聽眾每個 function 失靈在哪一步；(2) 對應到資通安全管理法 — 例如「資安事件應變程序」、「資通系統防護需求」這類條文層級；(3) 對應到 CIS Controls — 攻擊鏈每一段對應的 Control 編號；(4) 如果可能的話，對應到關鍵基礎設施保護指引或行政院技服中心 / 國家資通安全研究院（NICS）的相關情資通報範本。這對我們 GRC 圈的人是真正能放進公文的東西。

我也希望多一段「公部門 / 關基視角」的話 — 例如「如果這套被 nation-state APT 拿去用，國家級對策應該是什麼？CERT 通報？跨機關情資分享？國家資通安全會報的應變層級？」這場 talk 強調「速度差 30 倍是換維度」，但完全沒有提到我們國家層級的回應機制能不能跟得上。處長一定會問這個。

**少做：** 引擎內部的細節（slide 13–20）對我這種聽眾完全沒有 takeaway。如果是 50 分鐘的場次我可以接受，但 30 分鐘場次裡面塞 6–7 分鐘的 JSON / 公式 / Schema 對非工程聽眾是浪費。我會建議把這段濃縮成一張高層次的方塊圖，省下的時間給治理對應跟 Q&A。

AD 攻擊技術細節（slide 41–48）我也覺得太細。對我來說「AS-REP Roasting」、「ADCS ESC1」、「hashcat 離線爆破」三個名詞 + 一句結論「這些都不依賴弱密碼」就夠了。每個技術各佔兩張投影片是給紅隊聽眾的密度。

### 我會在我寫的年度報告引用的 3 個東西

**1. Slide 54 的 Mission Complete 四格數字（20 min / 3 of 3 / 0 人工介入 / 100% 信心可解釋）。** 引用文字會是：「2026 年 5 月 CYBERSEC 大會現場示範，國內紅隊團隊使用 AI 自主代理工具 Athena，於控制環境下 20 分鐘內全自動攻陷 3 台靶機，零人工介入。此資料顯示自主式 AI 攻擊工具已脫離學術概念階段，實作層級可運作於企業 AD 環境，事件應變時間目標應據此重新評估。」

**2. Slide 60 的三個 APT 案例（Storm-0558、Midnight Blizzard、Volt Typhoon）。** 直接抄成威脅 landscape 章節的子節，每個案例 1 段：攻擊者 → 入侵路徑 → 影響範圍。Volt Typhoon 那段會特別標註：「美國 CISA 評估與台海衝突相關，其 LOTL（Living off the Land）戰術強調長期潛伏，本國關鍵基礎設施機關應提高警覺，對應指引：關鍵基礎設施保護指引 / 資通安全管理法第 16 條。」

**3. Slide 61 的三個防守者問題（紅隊雲地同視 / SOC 同一張圖 / 事件應變速度）。** 我會把這三條翻成稽核問項，附在年度報告的「給機關的建議事項」章節：(1) 紅隊或滲透測試委外案是否同時涵蓋雲端與地端？(2) SOC 監控範圍是否涵蓋 AD、Entra ID、M365、雲端 KMS / Key Vault 的憑證與 token 流？(3) 事件應變 SLA 是否依當前自動化攻擊速度（20 分鐘量級）重新校準？這三條同時會是我下個月審 12 份下屬機關資安計畫書的核心稽核切入點。
